# Fact Check（ハルシネーション検出・除去）

## Overview

レポート草稿には、生成過程で紛れ込む「もっともらしい誤り」が必ず含まれ得る。本スキルは push 前の必須ゲートとして、草稿の全主張を検証し、出典に裏付けられない記述を除去・格下げする。

**大原則: 「リンクが生きていること」と「主張が正しいこと」は別物**

実証研究では、レポート中のリンク有効性は92〜96%と高い一方、事実正確性（主張が出典で裏付けられているか）は39〜77%にとどまる（出典: https://arxiv.org/html/2605.06635 ）。URLが200を返す・ページが実在する、というだけで合格にしてはならない。**主張の中身**を出典本文と突き合わせる。

## Iron Law

```
出典本文を読んで意味的に照合していない主張は、公開してはならない
```

## Step 1: クレーム台帳（Claim Ledger）作成

草稿を原子的主張（atomic claim = それ単独で真偽判定できる最小単位の文）に分解し、ID を付与する。

各クレームに種別タグを付ける：
- `数値`（%、金額、件数、順位など）
- `固有名詞`（社名、人名、製品名、地名）
- `日付`（発生時期、期限、期間）
- `因果`（「AのためB」「AによってB」）
- `評価`（「最大」「最先端」「唯一」等の強い形容）

各クレームに、根拠とする出典URLを対応付けた表を作る（Google DeepMind SAFE の分解方式に準拠: https://arxiv.org/abs/2403.18802 ）。

| ID | クレーム | 種別 | 出典URL |
|----|---------|------|---------|
| C001 | ○○社の売上は+15%成長 | 数値 | https://... |

## Step 2: 出典整合チェック（全クレーム）

各クレームの出典URLを WebFetch で取得し、本文と主張の意味的整合を確認する。以下の失敗パターンを名前付きでチェックする。

| パターン名 | 内容 | 処置 |
|---|---|---|
| 出典なし数値 | 出典に存在しない数値・%を記載 | 削除または定性表現に置換 |
| 幽霊引用 | 存在しないページ・著者への帰属 | 削除 |
| 引用ドリフト | 出典は実在するが主張内容がズレている | 主張を出典内容に合わせて修正 |
| 限定条件の脱落 | 「特定カテゴリ限定」「特定期間限定」を一般化して記載 | 限定条件を明記 |
| 循環引用 | 複数ソースに見えるが実は同一の一次情報の孫引き | 独立ソースとしてカウントしない |

## Step 3: 数値・固有名詞の triangulation

重要な数値・固有名詞は、独立した2ソース以上で一致を確認する。

- 2ソース以上一致 → そのまま採用
- 1ソースのみ → 確信度「低」を明記のうえ採用（本文にヘッジ語を付す）
- 複数ソース間で不一致 → 両論併記、または「未確定」として保留

## Step 4: CoVe式独立再検証

結論を支える上位5〜10件の最重要クレームについては、**草稿を見せない別サブエージェント**に検証質問だけを渡し、Web再調査させて回答を突合する（Chain-of-Verification: https://aclanthology.org/2024.findings-acl.212.pdf ）。

- 自分が書いた草稿を見せると、確認バイアス（自分の初稿を正当化する傾向）が働き検証にならない。必ず別コンテキスト・別サブエージェントで実施する。
- 渡すのは「クレームの真偽を問う質問」のみ。「この記述は正しいか？」ではなく「○○社の2025年売上成長率は何%か？」のように、答えを誘導しない形で問う。

## Step 5: 判定と処置

各クレームに以下いずれかを付与する。

- `SUPPORTED`: 出典が明確に裏付ける
- `PARTIAL`: 限定条件付きで支持される
- `UNSUPPORTED`: 出典で裏取りできない → 削除するか「推定」への格下げ＋ヘッジ語を付す
- `CONTRADICTED`: 出典と矛盾する → 修正必須（公開不可のまま放置しない）

## Step 6: 確信度ラベリング

主要な主張・数値には確信度を付与し、レポート本文に反映する。

- **高**: 独立2ソース以上が一致
- **中**: 1ソース＋傍証あり
- **低**: 単一ソースのみ、または推定

「わからない」と書くことは失敗ではない。裏取りできない場合はそう明記することが推奨される（Anthropic公式ガイダンス: https://platform.claude.com/docs/en/docs/test-and-evaluate/strengthen-guardrails/reduce-hallucinations ）。

## 検証ログの保存

`session/fact_check_log.json` に台帳と判定を保存する。

```json
{
  "checked_at": "2026-07-02T00:00:00Z",
  "claims": [
    {
      "id": "C001",
      "text": "○○社の売上は+15%成長",
      "type": "数値",
      "source_url": "https://example.com/report",
      "verdict": "SUPPORTED",
      "confidence": "高",
      "note": "2ソースで一致確認"
    }
  ]
}
```

## 過去の失敗事例（本リポ実録）

- **数値捏造**: 一次情報に存在しない成長率%（+268.6%等）を記載していた。出典本文には該当数値が存在せず、Step 2「出典なし数値」で検出すべき事例。
- **限定条件の脱落**: 「トップ収益者7/10が新規」という記述が、実際には特定カテゴリ限定のデータであるにもかかわらず、全体の話として一般化して記載されていた。Step 2「限定条件の脱落」で検出すべき事例。
- これらはいずれも Step 1〜2 のクレーム分解＋出典本文照合を徹底していれば防止できた。**URLが生きているかの確認だけでは検出できない**ことに注意。

## 注意: 検索深度を無闇に増やさない

ツールコール数を増やす（検索を深追いする）ほど事実精度が向上するとは限らず、むしろ最大42%低下するという報告がある。少数の高品質ソースを深く読むことを優先し、闇雲な追加検索でクレームを水増ししない。
