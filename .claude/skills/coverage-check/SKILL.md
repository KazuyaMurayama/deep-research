# Coverage Check（網羅性チェック）

## Overview

調査の偏り・重要視点の抜け漏れは、レポート公開後に「なぜこの視点がないのか」という批判を招く最大要因の一つである。本スキルは**研究設計時**と**公開前**の2回、異なる方法で網羅性を担保する。

## 適用タイミング

1. **研究設計時**（Agent1: research_architect）＝ 視点設計
2. **レポート起草後**（Agent4: quality_publisher の前）＝ 抜け漏れ検出

## 設計時ステップ1: 視点マトリクス生成

テーマに対し、次の軸で視点を列挙し、**調査対象に含める/含めない判断を明示的に記録**する（除外も理由付きで記録し、「検討したが除外した」ことを可視化する）。

- **ステークホルダー軸**: 当事者 / 提供者 / 規制当局 / 競合 / 批判者 / 周辺で影響を受ける人
- **時間軸**: 過去の経緯 / 現状 / 将来予測・変更予定
- **空間・スコープ軸**: 日本 / 海外、都市 / 地方、大手 / 中小 など
- **賛否軸**: 推進派の論拠 / 反対派の論拠 / 中立的評価
- **定量/定性軸**: 統計データ / 事例・体験談
- **PESTLE軸**（該当するもののみ）: 政治・法規制 / 経済 / 社会 / 技術 / 環境 / 法務

## 設計時ステップ2: 視点プロファイル別クエリ生成

検索クエリを単一視点のみで作らず、視点プロファイル（例: 規制当局視点 / 消費者視点 / 競合視点 / 技術者視点 / 懐疑派視点）ごとに個別生成する。同じ言い回しのクエリを量産する「クエリの均一化（redundant clustering）」を避け、視点ごとに異なる語彙・切り口で検索する。

## 設計時ステップ3: 反証探索クエリの必須化

各主要トピック・主要仮説に対し、「デメリット」「失敗」「批判」「リスク」「反対」系のクエリを**最低1本ずつ**設計に含める（Contrarian原則: https://arxiv.org/pdf/2605.20025 ）。肯定的な情報だけを集めて結論を補強する構成を避ける。

## 設計時ステップ4: breadth-first → narrowing

最初は短く広いクエリで全体像を把握し、そのあとで絞り込む2段階の検索を行う（Anthropic本番システムの原則: https://www.anthropic.com/engineering/multi-agent-research-system ）。いきなり狭いクエリから始めると、視野が偏ったまま調査が進んでしまう。

## 設計時ステップ5: premortem

「このレポートが完成後に『重要な視点が抜けていた』と批判されるとしたら、それは何についてか」を**3件以上**書き出す（Gary Klein のプリモーテム手法: https://hbr.org/2007/09/performing-a-project-premortem ）。書き出した各項目に対応するクエリ・セクションを設計に追加する。

## 公開前ステップ: completeness critic

レポート本文**だけ**を渡した独立サブエージェントに、次の問いのみを投げる。

> このレポートに欠けている観点・調査されていない領域・読者が当然期待するのに無い情報は何か？

- 指摘は親エージェントが裁定する。
- 重要な指摘 → 追加調査してレポートに反映
- 軽微な指摘 → 「本レポートの限界」セクションに明記して対応済みとする

## カバレッジ記録

`session.json` に次を記録し、Agent4 が公開前に照合できるようにする。

```json
{
  "coverage_matrix": {
    "included_perspectives": ["規制当局視点", "消費者視点", "..."],
    "excluded_perspectives": [
      { "perspective": "地方自治体視点", "reason": "本テーマのスコープ外と判断" }
    ]
  },
  "counter_evidence_queries": [
    "○○ デメリット",
    "○○ 失敗事例",
    "○○ 批判"
  ]
}
```
