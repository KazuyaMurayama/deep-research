# Independent Review（独立レビュー標準プロセス）

## Overview

自己採点だけでレポートを公開してはならない。生成した本人（同一セッション・同一コンテキストのモデル）は自分の出力を過大評価する傾向があり、公開前には必ず独立したレビュー工程を通す。

## 大原則1: 自己採点を公開スコアにしない

LLM は自己出力を実際より高く評価する傾向（self-preference bias）が知られている（出典: https://arxiv.org/html/2410.21819v2 ）。

**本リポ実録**: 自己申告92点のレポートが、独立レビューでの実測は86点だった。6点の乖離は自己採点の過大評価によるもの。**公開スコアは常に独立レビューの実測値を使う。**

## 大原則2: レビュアーへはレポート本文と出典のみを渡す

独立性を担保するため、レビュアーには次を**渡さない**：
- 生成過程のセッション履歴
- 検索履歴・中間ドラフト
- 「このレポートは高品質です」等の誘導情報

渡すのは完成した**レポート本文**と**引用出典一覧**のみ。生成コンテキストを共有しないサブエージェントを新規に起動する。

## 2段レビュー構成

### 第1段: 独立ファクトチェックレビュー

別サブエージェント（general-purpose）にレポート本文だけを渡し、Web再調査によるファクトチェックとルーブリック採点を行わせる。

**採点ルーブリック**（Anthropic本番Research機能のrubricに準拠: https://www.anthropic.com/engineering/multi-agent-research-system ）:

1. **factual accuracy** — 主張と出典の一致
2. **citation accuracy** — 引用と主張の整合
3. **completeness** — 要求観点の網羅
4. **source quality** — 一次情報優先か
5. **構造・So What明確性** — 論旨の明快さ、示唆の有無

### 第2段: 多観点パネルレビュー

テーマに応じて5〜7観点のレビュアーを**並列サブエージェント**で起動する。

- 観点例（テーマに応じて選定）: 戦略 / 批判的思考 / ドメイン専門家 / 読者代表 / 法務・リスク / ファイナンス 等
- **必ず1名は Contrarian（反証・弱点・交絡要因を専門に探す悪魔の代弁者）を含める**（DEBATE手法: https://openreview.net/forum?id=masq0lfC3s ）。Contrarian役には「このレポートの結論を崩す反証・見落とし・交絡要因だけを探せ」と明示的に指示する。

## 裁定プロセス

親エージェントが各指摘を **採用 / 部分採用 / 不採用** に裁定する。裁定基準：

1. **指摘自体を原典で検証してから反映する** — レビュアーも間違える。指摘を鵜呑みにして反映しない。
   - **本リポ実録**: あるレビュアー指摘は原典と矛盾していたため、検証の結果「不採用」が正解だった。レビューへの盲従は危険であり、指摘の裏取りを省略してはならない。
2. **ユーザーの目的・要件との整合を最優先する**
3. **迎合的な全面受け入れは禁止** — 指摘が多いほど良いレビューではない。的外れな指摘は明確に不採用とする。

## 公開スコアの記録

- 公開スコアは独立レビューの実測値を使用する。
- 自己採点と独立レビュー実測値が乖離した場合は、乖離幅も記録する（例: 自己申告92点 / 独立実測86点 / 乖離-6点）。

## レビュー結果の記録

レビュー結果は `tasks.md` に、各指摘の **採用/不採用の裁定理由**とともに記録する。次セッションが同じ指摘を再検証する手間を省くため、裁定根拠（原典のどの箇所で検証したか）まで残す。
