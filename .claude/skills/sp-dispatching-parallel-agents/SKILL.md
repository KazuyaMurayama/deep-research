# 並列エージェント起動（Dispatching Parallel Agents）

## Overview

独立したコンテキストを持つ専門エージェントにタスクを委譲する。指示とコンテキストを丁寧に設計することで、エージェントが脱線せずタスクに集中し成功する。エージェントは親セッションの文脈・履歴を継承しない — 必要な情報は親が明示的に構築して渡す。これにより親自身のコンテキストも温存される。

複数の無関係な失敗（異なるテストファイル、異なるサブシステム、異なるバグ）を逐次調査するのは時間の無駄。それぞれの調査が独立しているなら並列に行える。

**核心原則**: 独立した問題領域ごとに1エージェントを割り当て、並行して作業させる。

## 使うべき場面

- 3件以上の失敗・タスクが異なる根本原因を持つ
- 複数のサブシステムがそれぞれ独立して壊れている
- 各問題が他の文脈を見なくても理解できる
- 調査間で共有状態がない

## 使うべきでない場面

- 失敗が相互に関連している（1つの修正が他も直す可能性がある）
- システム全体の状態を理解する必要がある
- エージェント同士が干渉し合う（同じファイルを編集する、同じリソースを使う）

## パターン

### 1. 独立した領域を特定する

何が壊れているかで失敗をグルーピングする。各領域が独立していることを確認する（例: 領域Aの修正が領域Bに影響しない）。

### 2. 焦点を絞ったエージェントタスクを作る

各エージェントに与えるもの：
- **具体的なスコープ**: 1つのファイルまたはサブシステムのみ
- **明確なゴール**: 何を達成すれば完了か
- **制約**: 他のコードを変更しない、等
- **期待する出力**: 何を発見し何を修正したかの要約

### 3. 並列に起動する

同一メッセージ内で複数のサブエージェント起動を発行する — それらは並列に実行される。1メッセージにつき1件の起動は逐次実行になる。

### 4. レビューして統合する

エージェントが結果を返したら：
- 各要約を読む
- 修正同士が衝突していないか確認する
- 全体テスト/検証を実行する
- 全ての変更を統合する

## エージェントプロンプトの構造

良いエージェントプロンプトは次の3条件を満たす：
1. **焦点が絞られている** — 明確な1つの問題領域
2. **自己完結している** — 問題を理解するのに必要な文脈が全て含まれる
3. **出力が具体的** — 何を返すべきかが明示されている

## よくある間違い

| 悪い例 | 良い例 |
|---|---|
| 「全部直して」（スコープが広すぎる） | 「このファイルのこの失敗を直して」（焦点が明確） |
| 「競合状態を直して」（文脈なし） | エラーメッセージとテスト名を貼り付ける（文脈あり） |
| 制約なし（エージェントが無関係な範囲まで手を出す） | 「本番コードは変更しない」等の制約を明記 |
| 「直しておいて」（出力が曖昧） | 「根本原因と変更内容の要約を返して」（出力が具体的） |

## 使うべきでない場面（再掲）

- **関連する失敗**: 1つを直せば他も直る可能性がある → まず一緒に調査する
- **全体文脈が必要**: システム全体を見ないと理解できない
- **探索的デバッグ**: 何が壊れているかまだ分かっていない
- **共有状態**: エージェント同士が同じファイル・リソースで衝突する

## 主な利点

1. **並列化** — 複数の調査が同時に進む
2. **集中** — 各エージェントのスコープが狭く、追う文脈が少ない
3. **独立性** — エージェント同士が干渉しない
4. **速度** — 1件分の時間で複数件を解決できる

## 検証

エージェントが結果を返した後：
1. **各要約を読む** — 何が変わったかを理解する
2. **衝突を確認する** — 複数エージェントが同じコードを編集していないか
3. **全体検証を実行する** — 全修正が組み合わさって動くか確認する
4. **抜き取り確認する** — エージェントは組織的な誤りを犯すことがある

## 本リポ文脈への適用注記（並列Web調査サブエージェント）

本リポでは Agent2（web_researcher）等で複数の並列Web調査サブエージェントを起動する場面が多い。過去に、並列サブエージェントが「再帰的メタ応答」（自分自身の出力について再帰的に言及し続け、タスクの実行に進まない状態）に陥り、やむを得ず親が直接調査を引き取って切り替えた実例がある。

これを防ぐため、各サブエージェントには必ず次を含む**自己完結したプロンプト**を渡すこと：
- 背景（何を調べる調査か、なぜ必要か）
- 目的（この調査で明らかにすべき問いは何か）
- 出力形式（見出し・箇条書き等のフォーマット指定）
- 「ファイルを書かず、結果テキストをそのまま返答として返す」という明示的な指示

親セッションの会話履歴やメタな自己言及を誘発しうる文脈を渡さず、タスクそのものだけを渡すことが再帰的メタ応答の予防になる。
