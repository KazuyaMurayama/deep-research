## 第1章　エグゼクティブサマリー：クライアント有無で決まる最適解

### 最強の1本道（推奨構成を先出し）

**クライアント案件・商談・面接があるなら、迷わずこれを使え：**

```
VoiceMeeter Banana + OBS（ステレオ分離録音）
+ ffmpeg（チャネル分離）
+ WhisperX（個別文字起こし）
+ Claude Code 4エージェントパイプライン
```

この構成が「最強」である理由は4軸すべてを満たすからだ：

| 評価軸 | 評価 | 理由 |
|--------|------|------|
| クライアント体験 | ◎ | Bot通知なし・参加者リストに何も表示されない |
| 文字起こし精度 | ◎ | ffmpegチャネル分離→個別WhisperX処理により話者ラベルが100%確定 |
| 機密保持 | ◎ | 完全ローカル処理・音声がクラウドに出ない |
| Claude Code連携 | ◎ | 4エージェントパイプラインで議事録を自動生成 |

> **なぜ100%確定なのか：** OBSのステレオ録音で「左チャネル=自分の声」「右チャネル=相手の声」に分け、ffmpegで2つのWAVファイルに分離してから個別に文字起こしする。チャネルが完全に分かれているため、AI（機械学習）を使った話者推定が不要になる。

Day-1セットアップ所要時間：**約90〜120分**（Python環境構築含む）

---

### 3つの決定軸

どのツールを選ぶかは次の3軸で決まる。

**軸1：クライアント有無**
クライアント・商談・面接相手がいる場合、「Fireflies Note Taker が参加しました」という通知が相手に届く Bot型ツールは使わないほうがよい（詳細は第4章）。

**軸2：機密度**
社外秘・個人情報・M&A情報等を含む会議なら、音声データをクラウドに送らないローカル処理一択。

**軸3：運用コスト**
週1〜2回の会議なら無料枠のSaaSで十分。毎日複数本あるならローカル処理のコストパフォーマンスが高い。

## 第2章　推奨構成：VoiceMeeter Banana + OBS + ffmpeg + WhisperX

### なぜこれが最強か

従来の「OBS + Whisper」構成との最大の違いは **ステレオ分離録音 + ffmpegチャネル分離** にある。

```
[従来構成の問題点]
OBSが録音する音声ファイル：
  ├── モノラル or ステレオ（L=R、両方が混合）
  └── 話者分離は pyannote の機械学習に依存 → 精度85〜95%、エラーが発生しうる

[VoiceMeeter Banana + OBS + ffmpeg の優位性]
OBSが録音する音声ファイル：
  ├── Left チャネル = 自分のマイク音声（Razer Seiren Mini）
  └── Right チャネル = 相手のシステム音声（Zoom/Teams/Meet）
     ↓ ffmpegで分離
  mine.wav（左チャネルのみ） + other.wav（右チャネルのみ）
     ↓ 個別に WhisperX で文字起こし（--diarize 不要）
  → 話者ラベルが 100% 確定（機械学習不要・HuggingFaceトークン不要）
```

## 第3章　精度の最大レバー：音声チャネル分離

### 「ステレオ録音するだけ」では100%にならない

よくある誤解を先に正しておく。

**誤解：** 「OBSでステレオ録音すれば話者が100%分かれる」
**正解：** ステレオ録音は話者分離の前提条件だが、それだけでは不十分。WhisperX に `--diarize` フラグを付けて処理すると、内部でモノラルにダウンミックスされ、pyannote（AIによる話者推定）で処理されるため精度は85〜95%にとどまる。

**100%確定にする正しい手順：**
1. OBSでステレオ録音（左=自分、右=相手）
2. ffmpegでLチャネルとRチャネルを個別のWAVファイルに分離
3. mine.wav と other.wav をそれぞれ WhisperX で個別に文字起こし（`--diarize` 不要）

```powershell
# Step 1: ステレオMKVをL/R別WAVに分離
ffmpeg -i meeting.mkv -af "pan=mono|c0=c0" mine.wav
ffmpeg -i meeting.mkv -af "pan=mono|c0=c1" other.wav

# Step 2: 個別に文字起こし（--diarize は不要・HuggingFaceトークンも不要）
whisperx mine.wav  --model large-v2 --language ja --device cuda --compute_type int8_float16 --output_dir out
whisperx other.wav --model large-v2 --language ja --device cuda --compute_type int8_float16 --output_dir out
```

> ※ `-map_channel` は ffmpeg で非推奨化されたオプション。本ガイドでは公式推奨の `pan` フィルタを使用。

### large-v2 vs medium よりも効果的な理由

多くのガイドは「精度を上げるにはモデルを large-v2 に変えろ」と言う。それは正しいが、**話者分離精度という観点では音声チャネル分離のほうがはるかに効果が大きい**。

| 改善手段 | 話者ラベル精度 | 効果 |
|---------|---------------|------|
| medium → large-v2 に変更 | 変わらない（モデルは話者分離をしない） | 文字起こし精度+5〜15%程度 |
| モノラル→ステレオ録音のみ | 85〜95%（pyannote依存のまま） | 混入ノイズ減少で+5% |
| ステレオ録音＋チャネル分離＋個別transcribe | **100%（確定的）** | 機械学習不要 |
| pyannote（話者分離AI） | 85〜95% | モノラル音声での限界値 |

### 原理説明：なぜチャネル分離が確実なのか

通常の会議録音（モノラル/ステレオ混合）では、すべての音声が1つの信号に混ざっている。誰がいつ話したかを判定するには、音声の「声の特徴」を機械学習で分析する必要があり、これが誤認識の原因になる。

チャネル分離録音では、信号レベルで話者が分かれているため、機械学習が不要になる。

```
[チャネル分離なし → --diarize 使用]
time: 0:00  "[Lch+Rch混合信号]おはようございます..."
  → pyannote が確率的に推定 → 5〜15% の誤り

[チャネル分離あり → 個別transcribe]
mine.wav（Lch）: 0:00 "おはようございます"
other.wav（Rch）: 0:02 "よろしくお願いします"
  → WhisperX が確定的にラベリング → 誤り0%
```

### OBS でのステレオ分離設定方法

**方法1：パン設定（最も簡単）**

1. OBS を起動
2. 「音声ミキサー」パネルで「マイク」ソースを右クリック → 「詳細音声プロパティ」
3. 「マイク」行のパンを **完全に左（-100）** に設定
4. 「デスクトップ音声」行のパンを **完全に右（+100）** に設定
5. 出力設定で「ステレオ」を選択していることを確認

**方法2：マルチトラック録音（完全分離）**

1. OBS 設定 → 「出力」→「録画」→「出力モード：詳細」
2. 「音声トラック」でトラック1（マイク）とトラック2（システム音声）を分ける
3. ffmpeg で後処理（マルチトラックを1つのステレオに合成）：

```powershell
# トラック1（マイク）とトラック2（システム音声）を Left/Right に合成
ffmpeg -i meeting.mkv -filter_complex "[0:a:0]pan=stereo|c0=c0|c1=0[left];[0:a:1]pan=stereo|c0=0|c1=c0[right];[left][right]amix=inputs=2[out]" -map "[out]" meeting_stereo.wav
```

## 第4章　クライアント体験という見落とされた評価軸

### 「Note Taker が参加しました」が生む心理的摩擦

Bot型文字起こしツール（Fireflies.ai、Otter.ai、Notta Bot等）を使うと、会議参加者リストに次のような通知が表示される：

```
Fireflies Note Taker が会議に参加しました
Otter.ai Notetaker が会議に参加しました
```

この通知が相手（クライアント・採用担当者・商談相手）に届く。その心理的影響は：

- 「録音されている」という意識が生まれ、発言が慎重・形式的になる
- Botを許可しない会社・組織では入室が拒否される
- 初対面の商談・面接では「信頼関係構築」よりも「監視されている感覚」が先行する
- 重要な本音トークが引き出しにくくなる

### 2026年3月以降の Google Meet Bot 問題

2026年3月以降、**Google Meet では Bot の入室にホストの手動承認が毎回必要** になった。

これは tl;dv、Fireflies.ai、Notta Bot など Bot型ツール全般に影響する。

**具体的な運用コスト：**
- 会議開始前に Bot が「入室待機」状態になる
- ホスト（あなた）がスマートフォンまたはPC で「承認」をタップする必要がある
- 承認を忘れると Bot が入室できず、文字起こしが始まらない
- 外部ホストの会議（クライアント主催）では、ホストに承認を依頼しなければならない

Bot型ツールを Google Meet で使い続けることの運用コストは、2026年以降に大幅に増加している。

### Bot不参加型の戦略的価値

クライアント体験・精度・プライバシー・運用コストの4軸で考えると、**Bot不参加型**が総合的に優れている。

| ツール | Bot通知 | クライアント操作 | 推奨シナリオ |
|-------|---------|-------------|-----------|
| Granola（デスクトップ） | **なし** | 不要 | クライアント案件・商談・面接 |
| tl;dv（デスクトップモード） | **なし** | 不要 | クライアント案件・商談・面接 |
| OBS + ffmpeg + WhisperX（ローカル） | **なし** | 不要 | 機密会議・最高精度が必要な場合 |
| Zoom/Teams 内蔵 | 「文字起こし中」が全員表示 | なし | 社内MTG・全員同意済みの場合 |
| Bot型全般（Fireflies/Notta等） | **あり** | Meet：毎回手動承認（2026年3月〜）| 社内MTG・Bot承認済み環境のみ |

→ ツール別の総合比較は [第17章（付録B）](#第17章付録b-ツール比較表14ツール) を参照。

### Phase 1：VoiceMeeter Banana インストール・設定（20分）

VoiceMeeter Banana（ボイスミーターバナナ）は、複数の音声入力を混ぜてOBSに送るための「音声ルーター」アプリ。これを使うと「自分のマイクと相手の声を別々のチャネルに振り分ける」ことができる。

**Step 1：VoiceMeeter Banana をダウンロード・インストール（5分）**

1. https://vb-audio.com/Voicemeeter/banana.htm にアクセス
2. 「Download VoiceMeeter Banana」をクリック（無料、寄付推奨）
3. ダウンロードした `VoicemeeterProSetup.exe` を実行
4. インストール完了後、**必ずPCを再起動**（これをしないと音が出なくなる場合がある）

**Step 2：全デバイスのサンプルレートを 48kHz に統一（10分）**

> サンプルレート（音声データの細かさ）が揃っていないと、ピッチずれ・音声ノイズが発生する。必ず実施すること。

Windowsのサウンド設定を開く方法：タスクバーのスピーカーアイコンを右クリック → 「サウンドの設定」→ 下にスクロールして「詳細設定」

1. 「詳細設定」→「サウンドの詳細設定」→「録音」タブ
2. 「Razer Seiren Mini」を右クリック →「プロパティ」→「詳細」→「サンプルレート：48000Hz（DVD 品質）」
3. 「再生」タブ → 使用中のスピーカー/ヘッドフォンを右クリック → 同様に 48000Hz に設定
4. VoiceMeeter Banana 起動後、右上「System Rate」→「48000 Hz」に設定

**Step 3：VoiceMeeter Banana の入出力設定（5分）**

1. VoiceMeeter Banana を起動
2. `HARDWARE INPUT 1`（左上）の「Select Input Device」をクリック → 「WDM：Razer Seiren Mini」を選択
3. `HARDWARE OUT A1`（右上）の「Select Output Device」をクリック → 実際のスピーカー/ヘッドフォンを選択
4. Windowsの「サウンドの設定」→「出力デバイス」を「VoiceMeeter Input」に変更

VoiceMeeter Banana のボタン設定の確認：

| VoiceMeeterの入力欄 | A1ボタン（スピーカーで聴く） | B1ボタン（OBSが録音） |
|---|---|---|
| Hardware Input 1（マイク） | ON（緑に光らせる） | ON（緑に光らせる） |
| Virtual Input（PCの音） | ON（緑に光らせる） | ON（緑に光らせる） |

OBSの「音声出力キャプチャ」のデバイスとして `VoiceMeeter Output (VB-Audio VoiceMeeter VAIO)` を選ぶと、B1バス経由でマイクとPC音声の両方を録音できる。

### Phase 2：OBS インストール・ステレオ分離設定（20分）

OBS（オービーエス）は元々は配信・録画用ソフトだが、ここでは「マイクの音と相手の声を別々に録音する」ために使う。

**Step 1：OBS をダウンロード・インストール（5分）**

1. https://obsproject.com/ にアクセス
2. 「Windows」ボタンをクリック → インストーラーを実行
3. 自動構成ウィザードが表示されたら右下の「キャンセル」をクリック（表示されなければそのまま次へ）

**Step 2：出力設定（5分）**

「設定」→「出力」→「出力モード：詳細」に変更し、以下の通り設定する：

| 設定項目 | 値 |
|---|---|
| 出力モード | 詳細 |
| 録画フォーマット | mkv（クラッシュ時も保存されるため安全） |
| 音声チャンネル | ステレオ |
| 音声サンプルレート | 48 kHz |
| 録画パス | 例：`C:\Users\[ユーザー名]\Documents\MeetingRecordings` |

**Step 3：音声ソースを追加（5分）**

1. 「ソース」パネル → 「+」→「音声入力キャプチャ」→ 名前：`マイク`
2. デバイス：「Razer Seiren Mini」→ OK
3. 「+」→「音声出力キャプチャ」（PCから出ている音を録音する機能）→ 名前：`デスクトップ音声`
4. デバイス：「VoiceMeeter Output」または使用中のスピーカー → OK

> ※ OBSのバージョンにより「音声出力キャプチャ」に「WASAPI」の表記が付く場合がある。アプリ単位でキャプチャしたい場合は「アプリケーション音声キャプチャ（ベータ）」も使用できる。

**Step 4：ステレオ分離設定（5分）**

ここが最重要。自分の声を左、相手の声を右に振り分ける設定：

1. 「音声ミキサー」パネルで「マイク」ソースを右クリック →「詳細音声プロパティ」

### 7.5 文字起こし実行（バッチファイルで自動化）

毎回コマンドを打つのは手間なので、ドラッグ＆ドロップだけで使えるバッチファイルを作成する。

以下の内容を `transcribe.bat` という名前で保存（メモ帳に貼り付けて .bat 形式で保存）：

```batch
@echo off
chcp 65001

REM ===== 環境設定 (GPUなしの場合は以下3行を書き換える) =====
set MODEL=large-v2
set DEVICE=cuda
set COMPUTE=int8_float16
REM CPU版の場合: set MODEL=medium / set DEVICE=cpu / set COMPUTE=int8
REM =========================================================

set INPUT=%~1
set BASENAME=%~n1

if "%INPUT%"=="" (
    echo エラー: .mkv ファイルをこのバッチファイルにドラッグ&ドロップしてください
    pause
    exit /b 1
)

echo [1/4] チャネル分離中（自分の声と相手の声を分けています）...
ffmpeg -i "%INPUT%" -af "pan=mono|c0=c0" "%BASENAME%_mine.wav" -y
ffmpeg -i "%INPUT%" -af "pan=mono|c0=c1" "%BASENAME%_other.wav" -y

echo [2/4] 自分の声を文字起こし中...
whisperx "%BASENAME%_mine.wav" --model %MODEL% --language ja --device %DEVICE% --compute_type %COMPUTE% --output_dir "out_%BASENAME%"

echo [3/4] 相手の声を文字起こし中...
whisperx "%BASENAME%_other.wav" --model %MODEL% --language ja --device %DEVICE% --compute_type %COMPUTE% --output_dir "out_%BASENAME%"

echo [4/4] 完了しました。
echo 出力フォルダ: out_%BASENAME%
pause
```

**使い方：** 録画した .mkv ファイルをこのバッチファイルにドラッグ＆ドロップするだけ。

---

### 7.6 議事録の仕上げ

1. `out_xxxxx` フォルダ内の `_mine.srt`（自分の発言）と `_other.srt`（相手の発言）を開く
2. Claude Code に以下を依頼する：

```
mine.srtとother.srtをタイムスタンプ順にマージして、
mine=自分、other=相手として議事録を作ってください。
決定事項・アクションアイテム・未決事項を別ブロックで整理してください。
```

3. 誤認識の固有名詞を Ctrl+H で一括置換（例：「男座員」→「男座員也」）
4. 共有前に個人情報・機密情報の有無を確認

### 7.7 うまくいかないときの応急処置

| 症状 | 応急処置 |
|---|---|
| OBSの音声ミキサーのバーが片方だけしか動かない | 会議アプリ（Zoom等）の出力デバイスが「VoiceMeeter Input」になっているか確認 |
| OBSが録画できない | 出力先フォルダを `C:\Users\[自分の名前]\Documents\MeetingRecordings` に変更 |
| 文字起こしが途中で止まった | PowerShell を管理者権限で開いて再実行（`--output_dir` を `C:\out` などシンプルなパスに変更） |
| 急ぎで議事録が必要なのにWhisperXが遅い | Granola（バックアップ録音）の自動要約を先に使い、WhisperXは夜間バッチで処理 |

## 第8章　二重録音戦略

重要な会議には「プライマリ」＋「バックアップ」の二重録音を推奨する。

| 役割 | ツール | 目的 |
|------|--------|------|
| プライマリ | OBS + ffmpeg + WhisperX | 最高精度・ステレオ分離・ローカル保存 |
| バックアップ | Granola または tl;dv デスクトップ | OBS 録音失敗時の保険・簡易要約 |

**運用手順：**

1. 会議5分前に Granola を起動（バックアップとして常時動作）
2. OBS の「録画開始」ショートカットキーで録画開始（メイン録音）
3. 会議終了後、OBS 録画ファイルを transcribe.bat にドラッグ＆ドロップ
4. WhisperX 処理が終わるまでの間は Granola の自動要約を一時利用

**ショートカットキー設定（OBS）：**

OBS → 設定 → 「ホットキー」→「録画開始：Ctrl+Alt+R」「録画停止：Ctrl+Alt+S」に設定しておくと、会議中に画面を切り替えずに操作できる。

## 第9章　Claude Code 4エージェントパイプライン

WhisperX の出力テキストを Claude Code の4エージェントパイプラインに通すことで、高品質な議事録を自動生成する。

### パイプライン設計

```

  ↓

  ↓
Agent 2: Structurer（Sonnet）
  ↓
Agent 3: Decision/Action Extractor（Sonnet）
  ↓
Agent 4: Quality Reviewer（Opus）
  ↓
出力: 完成議事録（決定事項/ToDo/未決事項）
```

### Agent 1：Transcript Cleaner（Sonnet）

**役割：** フィラー除去、誤認識補正、固有名詞辞書による優先補正

## 第10章　Whisper 精度向上テクニック

### initial_prompt（固有名詞辞書）の使い方

Whisper は最初の数秒の認識精度が低い場合がある。`initial_prompt` に固有名詞・専門用語を渡すことで認識精度を向上させることができる。

```powershell
# 固有名詞辞書を initial_prompt で指定（WhisperX の場合）
whisperx mine.wav --model large-v2 --language ja --initial_prompt "プロジェクト名：DeepResearch、担当者：男座員也、クライアント：XXX株式会社、製品名：YYY" --device cuda --compute_type int8_float16 --output_dir out
```

### WhisperX 推奨コマンド集

```powershell
# GPU版（int8_float16 量子化でVRAMを節約しながら高精度を維持）
whisperx mine.wav --model large-v2 --language ja --device cuda --compute_type int8_float16 --output_dir out

# CPU版（int8 必須・mediumモデルが現実的）
whisperx mine.wav --model medium --language ja --device cpu --compute_type int8 --output_dir out
```

### Whisper モデル選択表

| モデル | 精度 | 処理速度 | VRAM目安 | 推奨用途 |
|-------|------|---------|---------|---------|
| tiny | 低 | 非常に速い | 1GB | 確認用 |
| base | 中 | 速い | 1GB | テスト用 |
| small | 中〜高 | 普通 | 2GB | 簡易用途 |
| **medium** | 高 | やや遅い | 5GB | **CPU 環境推奨** |
| **large-v2** | **最高** | 遅い | **推奨8GB**（int8量子化で6GB以下でも動作） | **GPU環境推奨** |

> WhisperX は large-v2 での動作が最も安定している（large-v3 は一部言語で精度が低下する場合がある）。

## 第11章　タイプレスの適用条件

Typeless（https://typeless.app/）は「音声入力ツール」であり、会議全体の文字起こしツールではない。

**Typeless が有効なシナリオ（英語/混合言語MTG限定）：**
- 英語または英日混合の会議で、自分のコメントをリアルタイムでドキュメントに入力したい
- 会議後に音声メモを素早くテキスト化したい（英語の発話のみ）
- 議事録の口述作成（会議後に録音した自分のまとめを文字化）

**Typeless を使うべきでないシナリオ：**
- **日本語純粋運用の会議**（日本語認識精度が English/Mixed より有意に劣る）
- 相手の声を含む会議全体の文字起こし（Typeless は自声のみ）
- 商談・面接・クライアントMTGの正式記録

> **結論：** 日本語のみの会議では Typeless の効果は薄い。英語または英日混合 MTG での補助ツールとして位置づけること。

## 第12章　セキュリティ・プライバシー指針

### 機密度別ツール選択フローチャート

```
会議に外部クライアントが含まれるか？
│
├── YES → Bot型ツールは使わない（Bot 通知が届く）
│   └── 推奨: OBS + ffmpeg + WhisperX（ローカル）または Granola
│
└── NO（社内MTGのみ）
    │
    └── 会議の機密度は？
        │
        ├── [非常に高い] 社外秘・個人情報・M&A 情報等
        │   └── OBS + ffmpeg + WhisperX ローカル処理一択
        │       ※音声・テキストが外部サービスに送信されない
        │
        ├── [中程度] 社内情報・プロジェクト情報
        │   └── 会社が SaaS 使用を認めているか？
        │       ├── YES → Notta / LINE WORKS AiNote（または内蔵）
        │       └── NO → OBS + ffmpeg + WhisperX ローカル処理
        │
        └── [低] 一般的な打ち合わせ・勉強会
            └── どのシナリオでも可（コスト・使いやすさで選択）
```

## 第13章　トラブルシューティング

### チャネル分離関連（ffmpeg）の問題

| 症状 | 原因 | 解決策 |
|------|------|--------|
| mine.wavとother.wavの両方に相手の声が入っている | OBSのパン設定が未実施またはVoiceMeeterのルーティングが誤り | OBSの詳細音声プロパティでマイクを左端（-100）、デスクトップを右端（+100）に設定し直す |
| ffmpegコマンドが見つからない | wingetインストール後にPowerShellを再起動していない | PowerShellを閉じて再度開いてから実行する |
| out_xxxフォルダにファイルが生成されない | 出力先フォルダの権限エラーまたはパスに日本語が含まれる | `--output_dir C:\out` のようにシンプルなパスを指定する |
| mine.wavに音が入っていない | OBSの左パン設定が効いていない | 録音フォーマットをmkvからwav(ステレオ)に変えてテストし直す |

### 音声関連の問題

| 症状 | 原因 | 解決策 |
|------|------|--------|
| 相手の声が録音されない | デスクトップ音声ソースが未設定 | OBSで「音声出力キャプチャ」を追加 |
| 自分の声が録音されない | マイクソースが未設定 or デバイスが違う | OBSの「音声入力キャプチャ」でRazer Seiren Miniを選択 |
| 音声が全く入らない | OBSの音声ミキサーがミュート | ミキサーパネルのミュートボタンを確認 |
| ヘッドフォンから音が出なくなった | VoiceMeeter 設定後に既定デバイスが変わった | タスクバー右クリック→サウンドの設定→出力デバイスを実際のヘッドフォンに戻す |
| 音声にピッチずれ・ノイズが発生 | サンプルレートが統一されていない | 全デバイスを 48kHz に統一（第6章 Phase 1 Step 2 参照） |
| ステレオ分離されていない | OBS のパン設定が未実施 | 詳細音声プロパティ → マイクを左端・デスクトップを右端に設定 |

### VoiceMeeter Banana 関連の問題

| 症状 | 原因 | 解決策 |
|------|------|--------|
| インストール後に音が出なくなった | Windows が VoiceMeeter を既定デバイスに設定した | サウンドの設定 → 出力を実際のスピーカー/ヘッドフォンに変更 |
| VoiceMeeter を起動するたびにデバイスが変わる | 起動順序の問題 | VoiceMeeter を Windows スタートアップに追加して自動起動させる |
| VoiceMeeter 経由で音質が劣化する | サンプルレート不一致 | 全デバイスとVoiceMeeterを 48kHz に統一 |
| PC 再起動後に設定がリセットされる | VoiceMeeter の設定保存が未実施 | Menu → Save Settings で設定ファイルを保存 |

### WhisperX 関連の問題

| 症状 | 原因 | 解決策 |
|------|------|--------|
| `whisperx` コマンドが見つからない | インストール失敗 or PATH 未設定 | `pip install whisperx` 再実行 → PowerShell 再起動 |
| VRAM 不足エラー | large-v2 モデルが GPU メモリ超過 | `--compute_type int8` を追加 or `--model medium` に変更 |
| 処理が非常に遅い | CPU のみで処理中 | GPU がある場合：CUDA をインストールし GPU 版 torch を導入 |
| 日本語が英語に変換される | 言語指定忘れ | `--language ja` オプションを必ず追加 |
| large-v2 で精度が低い | 音声品質の問題 | `--initial_prompt` で固有名詞を指定し認識精度を上げる |
| 処理が途中で止まる | メモリ不足 | `--model medium` に変更、または他のアプリを閉じてから実行 |

### WhisperX 再インストールコマンド

```powershell
pip uninstall whisperx -y
pip install whisperx
```

### Bot型 SaaS（Notta / AiNote）の問題

| 症状 | 原因 | 解決策 |
|------|------|--------|
| Bot が Google Meet に参加できない | 2026年3月〜の手動承認が必要 | 会議ホストが「Bot を承認」する必要がある（外部 MTG では相手に依頼が必要） |
| Bot が会議に参加しない | URL 入力ミス or 連携設定未完了 | 会議URLを再確認 → OAuthを再認証 |
| 文字起こしが日本語にならない | 言語設定が英語のまま | アカウント設定 → 言語 → 日本語に変更 |
| 無料枠が超過した | 月の利用時間を超えた | Granola（Bot不参加型）または OBS + WhisperX に切り替え |

## 第16章（付録A）　音声キャプチャ4方式の詳細

### 方式A：プラットフォーム内蔵文字起こし

会議プラットフォームのサーバーが全参加者の音声を処理するため、ローカルでの音声キャプチャが不要。

| 項目 | Zoom | Teams | Google Meet |
|------|------|-------|------------|
| 文字起こし機能 | AI Companion（有料） | 会議の文字起こし（有料） | Meet文字起こし（有料） |
| 必要プラン | Zoom Pro（$13.33/月・年払い） | M365 Business Basic以上 | Google Workspace有料 |
| 日本語対応 | 2024年から対応 | 対応 | 2025年3月から正式対応 |
| 話者分離 | あり | あり | あり |
| クライアント通知 | 「文字起こし中」が全参加者に表示 | 同左 | 同左 |

**推奨シナリオ：** 社内MTG・全参加者が同意している会議・既存プラン活用

**注意：** 「文字起こし中」通知は全参加者に表示される。外部クライアントが含まれる場合は要検討。

---

### 方式B：Bot型SaaS

文字起こしサービスのBotが会議に「参加者」として加わり、音声を録音・処理する。

**特徴：**
- 設定が最も簡単（10分以内）
- 日本語精度が高いサービスが多い
- **クライアントに Bot 通知が届く**
- **Google Meet：2026年3月以降、毎回ホストの手動承認が必要**

**推奨シナリオ：** 社内MTG・Bot 承認済みの組織内会議・精度より手軽さを優先する場合

---

### 方式C：Bot不参加型デスクトップアプリ

デスクトップアプリがシステムオーディオ＋マイクを直接キャプチャする。参加者リストに何も表示されない。

代表ツール：**Granola**、**tl;dv デスクトップモード**

**特徴：**
- クライアントへの通知なし
- プラットフォーム（Zoom/Teams/Meet）に依存しない
- システム音声を直接キャプチャするため Bot 承認問題が発生しない

**推奨シナリオ：** クライアント案件・商談・面接・Bot 通知を避けたい全ての会議

---

### 方式D：OBS + ローカルSTT（最推奨）

デスクトップ音声とマイク音声をOBSでステレオ分離録音し、ffmpegでチャネル分離後、WhisperX でオフライン文字起こし。

**特徴：**
- クライアントへの通知なし
- 完全ローカル処理（音声データがクラウドに出ない）
- ffmpegチャネル分離による話者ラベル100%確定
- Claude Code 4エージェントパイプラインと最適に連携
- 完全無料・Zoom/Teams/Meet問わず対応

**推奨シナリオ：** 機密会議・クライアント案件・最高精度を求める場合・プライバシー最優先
